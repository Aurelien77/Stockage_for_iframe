Chargement + scan des 3 images
📋 copier
# 1. Charger les images depuis les artifacts du stage package
docker load < front.tar.gz
docker load < back.tar.gz
docker load < standalone.tar.gz
 
# 2. Générer le rapport HTML pour chaque image
trivy image --format template --template @html.tpl \
--output trivy-back.html microcrm-back:$CI_COMMIT_SHA
 
# 3. Exit 1 si CVE CRITICAL trouvée (→ badge orange)
trivy image --exit-code 1 --severity CRITICAL \
microcrm-back:$CI_COMMIT_SHA
🟠 Logique orange — pourquoi le pipeline ne bloque pas
⚙️
allow_failure: true dans le .gitlab-ci.yml
Même si Trivy retourne un exit 1 (CVE CRITICAL détectée), le job est marqué en warning et le pipeline continue vers le stage publish.
🎨
Résultat visuel dans GitLab : badge orange ⚠️
Le pipeline affiche un badge orange (pas rouge). L'équipe voit immédiatement qu'il y a une CVE CRITICAL à traiter, sans que cela bloque la livraison en urgence.
🤔
Pourquoi ce choix ?
Certaines CVE CRITICAL concernent des bibliothèques système non exposées (ex : libssl dans l'image de base). L'équipe Ops décide au cas par cas si la vulnérabilité est réellement exploitable dans notre contexte.
💡 Points intéressants à noter
🔄
Scan AVANT le publish — pas après
Contrairement à l'ancien workflow où Trivy était appliqué par l'équipe Ops après pull, il est maintenant exécuté dans le CI avant tout push au registry GitLab. Les CVE sont détectées dès la construction.
📊
3 rapports HTML téléchargeables
Trivy génère un rapport .html interactif par image, disponible en artifacts GitLab. Chaque rapport liste les CVE avec leur description, score CVSS et correctif disponible.
🗄️
Base de données CVE auto-mise à jour
Trivy télécharge sa DB depuis ghcr.io/aquasecurity/trivy-db à chaque exécution. Le scan utilise toujours les vulnérabilités les plus récentes connues.
🧅
Scan par couche Docker
Trivy analyse chaque couche de l'image : l'OS de base, les packages système, les dépendances Java (JAR), les modules npm Angular. Les CVE sont attribuées à la couche précise concernée.
